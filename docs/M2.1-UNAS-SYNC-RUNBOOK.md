# M2.1 UNAS Product Sync – migration és recovery runbook

## Cél

Az M2.1 adatbázis-migráció, a terméktükör és a sync lifecycle ellenőrzése
izolált PostgreSQL adatbázison. A teszt nem hív valódi UNAS API-t, és nem igényel
UNAS API-kulcsot.

## Előfeltételek

- futó PostgreSQL, kizárólag tesztelésre létrehozott üres adatbázissal;
- a `DATABASE_URL` a lokális, gitignore-olt `.env` fájlban;
- telepített workspace dependency-k;
- production adatbázis használata tilos.

Példa, kizárólag lokális fejlesztői környezethez:

```env
DATABASE_URL=postgresql://acropora:local-password@localhost:5432/acropora_m21_test?schema=public
```

## Migráció

```bash
pnpm prisma:generate
pnpm prisma:deploy
pnpm db:validate
```

A `prisma:deploy` után a migrációs státusznak eltérés nélkül kell lefutnia:

```bash
pnpm --filter @acropora/database exec prisma migrate status
```

## Integrációs tesztek

```bash
pnpm --filter @acropora/api test:integration
```

Az M2.1 Product Sync teszt ellenőrzi:

1. első teljes mirror létrehozását;
2. azonos payload idempotens újrafuttatását;
3. SKU módosítását stabil UNAS Product ID mellett;
4. Product Extension megőrzését szinkron közben;
5. teljes snapshot hiányából a `MISSING` állapotot;
6. változatlan payload visszatérésekor a restore lifecycle-t;
7. adatbázis-szintű párhuzamos futás kizárását;
8. hogy nem keletkezik `StockMovement` vagy `StockItem` rekord.

## Read-only UNAS API probe

A probe nem használ Prisma klienst, nem olvas és nem ír adatbázist, és kizárólag
`login`, `getCategory`, valamint külön live/deleted `getProduct` hívásokat végez.
Ne add át az API-kulcsot CLI argumentumban vagy parancssori környezeti
változóként. A package script a repository gyökerében lévő, gitignore-olt `.env`
fájlt a Node `--env-file` kapcsolójával tölti be:

```bash
pnpm --filter @acropora/api unas:probe
```

- `--page-size`: alapérték 10, megengedett tartomány 1–100;
- `--pages`: alapérték 1, megengedett tartomány 1–2, adathalmazonként értendő;
- a stdout csak aggregált strukturális JSON összesítést tartalmaz;
- hiba esetén a stderr csak `{ "ok": false, "errorCode": "..." }` alakú
  normalizált választ kap, és a process nem nulla kóddal áll le;
- hiányzó `UNAS_API_KEY` esetén az error code
  `UNAS_PROBE_API_KEY_MISSING`;
- a kimenetet ne egészítsd ki debug HTTP vagy raw payload naplózással.

A futási hibák allowlist-alapú, stage-specifikus kódot kapnak
`UNAS_PROBE_<LOGIN|CATEGORY|LIVE|DELETED>_<REASON>` formában. A biztonságos
reason értékek külön jelzik többek között a jogosultság, autentikáció, rate
limit, HTTP 4xx/5xx, timeout, hálózat, UNAS API elutasítás, hibás vagy túl nagy
XML, response-shape és mezőformátum problémákat. A kód soha nem tartalmazza az
eredeti kivételüzenetet, az UNAS `Error` szövegét vagy a response body tartalmát.
Ha a login válasz dokumentált `Permissions/Permission` struktúrája felismerhető,
a probe a `getCategory` és `getProduct` jogokat a hívások előtt ellenőrzi;
ismeretlen struktúra nem minősül hiányzó jogosultságnak.

A probe eredménye nem indít syncet, apply-t vagy migrációt. A célzott lokális
teszt API-kulcs és hálózat nélkül fut:

```bash
pnpm --filter @acropora/api test:unas:probe
```

Az M2.2 connection settings nem változtatja meg a probe credential-forrását: a
probe továbbra is kizárólag a gyökér, gitignore-olt `.env` fájlból betöltött
`UNAS_API_KEY` értéket használja, és nem olvas adatbázis credentialt.

## M2.2 connection settings üzemeltetés

Az API indítása előtt legyen beállítva az aktív kulcsverzió és a hozzá tartozó,
32 véletlen bájtot tartalmazó base64 főkulcs:

```text
UNAS_CREDENTIAL_ACTIVE_KEY_VERSION=1
UNAS_CREDENTIAL_MASTER_KEY_V1=<base64-encoded-32-byte-key>
```

A főkulcsot ne add át CLI argumentumban és ne írd shell parancssorba. Production
secret store-ból vagy védett környezeti konfigurációból kerüljön a processhez.

A credential módok pontos jelentése:

- `ENV_FALLBACK`: átmenetileg az `UNAS_API_KEY` használható;
- `DATABASE`: csak a titkosított adatbázis credential használható; hiba esetén
  nincs env fallback;
- `DISABLED`: nincs credential, az env fallback akkor is tilos, ha be van állítva.

Hiányzó singleton nem jelent implicit env fallbacket: az API és a szinkron
`UNAS_CONNECTION_CONFIGURATION_MISSING` hibával fail-closed áll le. A migráció
deterministikusan létrehozza az `unas` rekordot. Production startup ellenőrzi a
rekordot, az aktív írási főkulcsot, DATABASE módban pedig a hivatkozott
kulcsverziót és az envelope hitelesített visszafejthetőségét. Hibánál az API nem
jelentkezik egészségesként. ENV_FALLBACK módban az `UNAS_API_KEY` megléte is
startup-feltétel; DISABLED módban credential nélkül indul, de szinkront nem végez.

Új credential mentése előbb read-only logint és ismert permission struktúránál
`getProduct`/`getCategory` ellenőrzést végez. Ismeretlen permission struktúra
`INDETERMINATE` figyelmeztetés, biztosan hiányzó jog elutasítás. A tárolt kulcs
külön tesztelhető; az UNAS-t ténylegesen elérő manuális teszt auditált. Cooldown
miatt elutasított kérés nem hoz létre audit rekordot. A cooldown PostgreSQL
`CURRENT_TIMESTAMP` alapján atomikus. A 24 óránál régebbi ellenőrzés `STALE`, de
önmagában nem állítja le a szinkront. Credentialcsere vagy disable közben elavulttá
váló teszteredmény nem írja felül az aktuális verification állapotot; auditja
`STALE_TEST_RESULT`.

Az UNAS login dokumentált `Expire` (kompatibilitásként `ExpireTime`) mezője UNIX
timestamp, a dokumentált API-key token
élettartama két óra. Lejárt, nulla, egy percnél rövidebb hátralévő idejű vagy a
két órát öt percnél jobban meghaladó válasz `UNAS_CONNECTION_RESPONSE_INVALID`.

Kulcsrotációnál előbb add hozzá az új verzió környezeti főkulcsát, utána állítsd
aktívra. A régi főkulcsot csak minden envelope újratitkosítása és a rollbackablak
lezárása után távolítsd el. Az adatbázis credentialt tilos visszafejteni és
exportálni rollback céljából.

Rollbacknél az additív singleton táblát ne töröld. Korábbi alkalmazásverzióra
visszaálláskor a verziózott főkulcsokat és az aktív kulcsverziót őrizd meg;
DATABASE módból csak explicit, auditált disable vagy új credential mentés váltson.

## Sikertelen futás vizsgálata

1. Nyisd meg az `/admin/integrations/unas` oldalt.
2. Jegyezd fel a run ID-t és a biztonságos `errorCode` értéket.
3. Ellenőrizd, hogy a futás `FAILED`, és nincs aktív futáskulcsa.
4. Ellenőrizd, hogy az integration cursor nem lépett előre.
5. Az ok javítása után indíts kézi szinkront. Sikertelen futást ne módosíts kézzel
   `APPLIED` állapotra.

Beragadt futásnál a heartbeat nélküli rekord 15 perc után a következő indításkor
`UNAS_PRODUCT_SYNC_STALE` hibával lezáródik. Az aktív kulcs kézi törlése csak
adatbázis-mentés után és dokumentált incidenskezelés részeként megengedett.

## Rollback

- alkalmazáskód rollbackje előtt ellenőrizni kell, hogy a korábbi verzió kezeli-e
  az új nullable oszlopokat és táblákat;
- az M2.1 migráció automatikus lefelé migrációt nem tartalmaz;
- adatvesztéssel járó séma-visszaállítás helyett alkalmazás rollback és forward
  fix az elsődleges eljárás;
- `ProductExtension`, `UnasProductSnapshot`, receipt vagy inventory történet nem
  törölhető recovery céljából.

## Éles aktiválás

Az integrációs tesztek után is kikapcsolva marad az időzítés:

```env
UNAS_PRODUCT_SYNC_ENABLED=false
```

Az első éles futás kézzel, felügyelet mellett történik. Csak az eredmény és a
reported stock eltérések ellenőrzése után kapcsolható be az ütemező.
