# M2.1 UNAS Product Sync – migration és recovery runbook

## Cél

Az M2.1 adatbázis-migráció, a terméktükör és a sync lifecycle ellenőrzése
izolált PostgreSQL adatbázison. A teszt nem hív valódi UNAS API-t, és nem igényel
UNAS API-kulcsot.

## Előfeltételek

- futó PostgreSQL, kizárólag tesztelésre létrehozott üres adatbázissal;
- a `DATABASE_URL` a lokális, gitignore-olt `.env` fájlban;
- telepített workspace dependency-k;
- production adatbázis használata tilos.

Példa, kizárólag lokális fejlesztői környezethez:

```env
DATABASE_URL=postgresql://acropora:local-password@localhost:5432/acropora_m21_test?schema=public
```

## Migráció

```bash
pnpm prisma:generate
pnpm prisma:deploy
pnpm db:validate
```

A `prisma:deploy` után a migrációs státusznak eltérés nélkül kell lefutnia:

```bash
pnpm --filter @acropora/database exec prisma migrate status
```

## Integrációs tesztek

```bash
pnpm --filter @acropora/api test:integration
```

Az M2.1 Product Sync teszt ellenőrzi:

1. első teljes mirror létrehozását;
2. azonos payload idempotens újrafuttatását;
3. SKU módosítását stabil UNAS Product ID mellett;
4. Product Extension megőrzését szinkron közben;
5. teljes snapshot hiányából a `MISSING` állapotot;
6. változatlan payload visszatérésekor a restore lifecycle-t;
7. adatbázis-szintű párhuzamos futás kizárását;
8. hogy nem keletkezik `StockMovement` vagy `StockItem` rekord.

## Sikertelen futás vizsgálata

1. Nyisd meg az `/admin/integrations/unas` oldalt.
2. Jegyezd fel a run ID-t és a biztonságos `errorCode` értéket.
3. Ellenőrizd, hogy a futás `FAILED`, és nincs aktív futáskulcsa.
4. Ellenőrizd, hogy az integration cursor nem lépett előre.
5. Az ok javítása után indíts kézi szinkront. Sikertelen futást ne módosíts kézzel
   `APPLIED` állapotra.

Beragadt futásnál a heartbeat nélküli rekord 15 perc után a következő indításkor
`UNAS_PRODUCT_SYNC_STALE` hibával lezáródik. Az aktív kulcs kézi törlése csak
adatbázis-mentés után és dokumentált incidenskezelés részeként megengedett.

## Rollback

- alkalmazáskód rollbackje előtt ellenőrizni kell, hogy a korábbi verzió kezeli-e
  az új nullable oszlopokat és táblákat;
- az M2.1 migráció automatikus lefelé migrációt nem tartalmaz;
- adatvesztéssel járó séma-visszaállítás helyett alkalmazás rollback és forward
  fix az elsődleges eljárás;
- `ProductExtension`, `UnasProductSnapshot`, receipt vagy inventory történet nem
  törölhető recovery céljából.

## Éles aktiválás

Az integrációs tesztek után is kikapcsolva marad az időzítés:

```env
UNAS_PRODUCT_SYNC_ENABLED=false
```

Az első éles futás kézzel, felügyelet mellett történik. Csak az eredmény és a
reported stock eltérések ellenőrzése után kapcsolható be az ütemező.
