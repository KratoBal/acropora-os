# M2.1 – UNAS API discovery

Ellenőrzés dátuma: 2026-07-20

## Eredmény

Az UNAS hivatalos API-dokumentációja alapján az M2.1 API-alapú Product Mirror
megvalósítható. A discovery dokumentációs szerződést igazol; éles shopon végzett
probe továbbra is szükséges az előfizetés, jogosultság, tényleges válaszméret és
rate limit ellenőrzéséhez.

## Igazolt szerződés

| Kérdés              | Eredmény                                                    | M2.1 döntés                                         |
| ------------------- | ----------------------------------------------------------- | --------------------------------------------------- |
| Stabil product ID   | Az `Id` integer értéket az UNAS osztja ki létrehozáskor     | elsődleges `ExternalReference.externalId`           |
| SKU módosítható?    | Igen, a `SkuNew` külön SET mező; a history régi SKU-t is ad | SKU nem technikai identity                          |
| Módosítás időpontja | `LastModTime`, Unix timestamp                               | inkrementális cursor bemenet                        |
| Időablakos lekérés  | `TimeStart` és `TimeEnd`                                    | zárt futási időablak átfedéssel                     |
| Lapozás             | `LimitStart` + `LimitNum`                                   | offset pagination, stabil deduplikáció ID alapján   |
| Törlés              | `State=deleted`, egy hónapra visszamenőleg                  | rendszeres full reconciliation továbbra is kötelező |
| Termékstátusz       | 0 inaktív; 1 aktív; 2 aktív/új; 3 aktív/nem vásárolható     | raw érték mellett igazolt display mapping           |
| Készletlekérés      | külön `getStock`, Id/SKU és lapozás támogatott              | reported-stock snapshot                             |
| Készlet inkrementum | `getStock.TimeStart` csak mozgással érintett terméket ad    | hiány nem jelent nulla készletet                    |
| Protokoll           | POST + XML, TLS 1.2/1.3                                     | szerveroldali adapter és szigorú XML parser         |

## Cursor és lapozási stratégia

Egy inkrementális futás induláskor rögzíti a `windowEnd` időpontot, majd
`TimeStart=previousSuccessfulWindowEnd-overlap` és `TimeEnd=windowEnd`
paraméterekkel kérdez. Az overlap kezdetben 120 másodperc. A lapok
`LimitStart/LimitNum` használatával töltődnek le, a feldolgozás UNAS `Id` alapján
deduplikál.

A cursor csak akkor lép `windowEnd` értékre, ha minden oldal sikeresen
feldolgozódott és az adatbázis apply commitolt. Részleges futás nem mozdítja a
high-water markot.

Mivel az offset-lista a lapozás közben változhat, időszakos teljes
reconciliation szükséges. A teljes futás is ID alapján deduplikál, és csak ez
jelölhet `MISSING` terméket.

## Identity és SKU-változás

Az `ExternalReference(system=UNAS, entityType=Product, externalId=<Id>)` az
elsődleges feloldás. Az API history `modify` eseménye `SkuOld` és aktuális `Sku`
értéket adhat. SKU-változás ugyanazt a `ProductVariant` rekordot frissíti, ha az
UNAS ID-egyezés egyértelmű és az új SKU nincs más varianthez rendelve.

Ha az UNAS ID és az SKU két külön belső rekordra mutat, az eredmény `CONFLICT`;
a rendszer nem merge-el automatikusan.

## Készletszemantika

A `getStock.TimeStart` dokumentáltan csak azokat a termékeket adja vissza,
amelyeknél az időszakban készletmozgás történt. Ezért:

- hiányzó rekord nem jelent nulla készletet;
- a válasz csak `UnasProductSnapshot.reportedStock` értéket frissít;
- a product mirror futás nem írhat `StockItem` vagy `StockMovement` adatot;
- rendszeres teljes getStock reconciliation szükséges a snapshot teljességéhez.

## Éles probe előtt szükséges

- UNAS PREMIUM vagy VIP csomag és API-hozzáférés igazolása;
- read-only scope/token használata, ha a shop konfiguráció ezt támogatja;
- teszttermék ID és SKU;
- egy minimális, egy időablakos és egy lapozott `getProduct` válasz rögzítése
  secretmentes fixture-ként;
- egy teljes és egy `TimeStart` alapú `getStock` válasz szemantikájának mérése;
- tényleges rate-limit és maximális biztonságos `LimitNum` meghatározása;
- shop alapdevizájának igazolása.

## Implementációs állapot

Elkészült:

- API-key login és Bearer-tokenes `getProduct` kliens;
- SAX-alapú, DTD/entity tiltásos XML parser;
- `TimeStart/TimeEnd` és `LimitStart/LimitNum` lapozás;
- 120 másodperces cursor overlap és ID-alapú oldaldeduplikáció;
- canonical SHA-256 payload hash;
- external ID elsődleges és SKU másodlagos identity diff;
- perzisztens sync run és high-water mark cursor;
- serializable mirror apply, ahol a cursor csak a domainírásokkal azonos sikeres
  tranzakcióban lép előre.
- dokumentált API mapping névhez, leírásokhoz, árakhoz, áfához, kategória-ID-khez,
  képekhez, paraméterekhez, mennyiségi szabályokhoz, mértékegységekhez,
  stock-beállításokhoz, reported stockhoz, URL-ekhez és SEO-adatokhoz;
- teljes snapshot `MISSING` jelölés, újramegjelenéskor `RESTORE`, fizikai törlés
  vagy automatikus archiválás nélkül;
- legacy SKU-alapú Product external reference átvezetése valódi UNAS ID-ra.
- lapozott `getCategory` parser és kétmenetes Category/parent reconciliation;
- normalizált UNAS `ProductCategory` és `ProductImage` újraszinkronizálás úgy,
  hogy a manuális forrású kapcsolatok és képek megmaradnak;
- külön `State=deleted` terméklekérés, amely az érintett mirror rekordot
  `MISSING` állapotba teszi.

Még nem kapcsolható automatikus production futásra:

- nincs credential konfiguráció és admin trigger;
- nincs éles shop probe;
- az XLSX „Menny. kezdeti érték” és alapár-pénznem mezőjéhez nincs igazolt
  Product API node; ezek éles probe-ot vagy külön setting lekérést igényelnek;
- a törölt kategóriák retention policy-ja még nincs meghatározva; ezért azok
  automatikusan nem törlődnek és nem archiválódnak.
